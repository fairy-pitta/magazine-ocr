## 現状の問題サマリー

最新の全画像実行結果 (`dev-sample-final-b`) から判明した問題:

| 問題 | 深刻度 | 影響範囲 |
|---|---|---|
| 都道府県が全レコード `null` | 致命的 | 全45レコード |
| シリアル番号がほぼデタラメ | 致命的 | 連番性が全く無い |
| OCRテキストが大半ゴミ文字 | 致命的 | location_raw が判読不能 |
| 行検出の精度が不安定 | 高 | 一部ページでフォールバック5等分 |
| ページ番号は概ね正しい | — | 31,36-43 を正しく取得 |

根本原因: **手撮り写真に対してTesseractの縦書き日本語OCRが機能していない**。
前処理が不足しており、行のクロップ精度も低いため、OCRに渡る画像の品質が悪い。

---

## 戦略方針: Claude Code (サブスク内) を最大活用する

5画像・約50行クロップという小規模データなので、Claude Code の画像読み取り機能を
API セットアップなしで直接活用できる。

**Claude Code でやること:**
1. 正解データの作成 — 画像を直接読ませてアノテーション生成
2. OCR精度の比較実験 — 行クロップを読ませて Tesseract と比較
3. AI Review — 低信頼度ケースの判定

**将来データが増えた場合のみ** Cloud Vision API や Claude API のバッチ自動化を検討する。

## Phase 4: OCR戦略の根本的見直し

**現状の最大のボトルネック。** Tesseractの縦書き日本語OCRが手撮り写真にほぼ機能していない。

### 4-1. アプローチの選択肢を検証

以下の選択肢を **IMG_8799 の1画像** で比較実験する:

- [ ] **選択肢A: Tesseractのチューニング**
  - 前処理改善 (Phase 2) + PSMモード変更 + 文字ホワイトリスト調整
  - 行クロップを回転して横書きとしてOCR (`--psm 6`)
  - 画像拡大率を変えて最適値を探す (現在2x→3x, 4xも試す)

- [ ] **選択肢B: Cloud Vision API に切り替え**
  - Google Cloud Vision API は手撮り日本語に強い
  - 行クロップ単位で送信すればコストは低い (5画像×10行×2ページ = 100リクエスト程度)
  - `DOCUMENT_TEXT_DETECTION` で縦書き対応

- [ ] **選択肢C: Claude Code (サブスク内) で対話的に処理**
  - 行クロップ画像を Claude Code の Read ツールで読み込み
  - 「この行画像からシリアル番号と都道府県を抽出せよ」の指示で構造化抽出
  - 追加コストゼロ、API Key セットアップ不要
  - 5画像規模なら対話的に完結可能
  - データが増えた場合は Claude API バッチに移行

- [ ] **選択肢D: Claude API でバッチ自動化**
  - 選択肢Cが有効と確認できた場合、自動化のために API 化
  - docs/architecture.md の "AI Review Layer" 設計と整合する
  - データ規模が大きくなった時にのみ必要

- [ ] 比較結果を `docs/ocr-comparison.md` に記録

### 4-2. 推奨判断基準

- Tesseractチューニングで recall 70%以上 → Tesseract続行
- それ未満 → Claude Code で対話的に処理（5画像規模）or Cloud Vision に移行
- 5画像で収まるなら Claude Code (サブスク内) が最もコスパ良い
- 将来データが増える場合は Cloud Vision か Claude API に移行

### 4-3. シリアル番号OCRの改善

OCRエンジンの選択とは別に、アンカー領域の抽出を改善する:

- [ ] ☆マークのテンプレートマッチングまたは形状検出で位置を特定
- [ ] ☆の右隣の数字部分だけをクロップしてOCR（現在は行の右半分全体をOCR）
- [ ] 数字のみのホワイトリスト (`0123456789`) でOCR精度を上げる

### 4-4. 都道府県ヘッダーOCRの改善

全レコードで `null` になっている最大の問題:

- [ ] 都道府県ヘッダーの視覚的特徴を特定（太字？括弧付き？独立行？）
- [ ] ヘッダー領域の検出ロジックを追加（現在は行内の左半分をOCRしているだけ）
- [ ] 47都道府県リストとの fuzzy matching を導入
- [ ] OCRが失敗しても、行の視覚的特徴（短いテキスト行、太字など）でヘッダー候補を検出

## 推奨実行順序

```
Phase 1 (正解データ)  ← まずここ。IMG_8799だけでよい
    ↓
Phase 2 (前処理改善)  ← 画像品質を上げる
    ↓
Phase 4 (OCR戦略見直し) ← Tesseract継続 or 乗り換えを判断
    ↓
Phase 3 (行検出改善)  ← 前処理改善後に再評価
    ↓
Phase 5 (連番修復)    ← OCRがまともになってから
    ↓
Phase 6 (AI Review)   ← 決定論的パイプラインが安定してから
    ↓
Phase 7-8 (レポート・バリデーション)
    ↓
Phase 9 (仕上げ)
```

**最初の判断ポイント**: Phase 1 で Claude Code に IMG_8799 を読ませた時点で、
Claude の画像読み取り精度が判明する。高精度なら Tesseract 改善は後回しにして
Claude Code ベースで全画像を処理するのが最短ルート。
